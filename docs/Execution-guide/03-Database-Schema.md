# CaaS Platform — Database Schema Blueprint
### Exact Table Definitions | Version 1.0 | 26 May 2026

---

## Conventions

- All primary keys are strings using the format `prefix_<hex>` (e.g., `usr_abc123`) generated by `'prefix_' + crypto.randomBytes(12).toString('hex')`. UUIDs are used where already established (e.g., `users.id`).
- All timestamps are ISO 8601 strings stored as `TEXT` (SQLite has no native timestamp type). Always stored in UTC.
- `tenant_id` appears on every table that holds tenant data. Every query against these tables MUST include `WHERE tenant_id = ?`. No exceptions.
- Foreign key enforcement: `PRAGMA foreign_keys = ON` set at connection time.
- The schema is additive — existing tables (`users`, `accounts`, `user_profiles`) are extended with new columns via `ALTER TABLE ... ADD COLUMN`, never recreated.

### `accounts` — Tenant registry

```sql
CREATE TABLE accounts (
  id              TEXT PRIMARY KEY,
  tenant_id       TEXT NOT NULL UNIQUE,
  tier            TEXT NOT NULL CHECK(tier IN ('LITE', 'GROWTH', 'ENTERPRISE')),
  status          TEXT NOT NULL CHECK(status IN ('pilot', 'active', 'suspended', 'churned'))
```

## New Tables — MVP Extensions

### `scan_events` — Incoming AI decision events

```sql
CREATE TABLE scan_events (
  id              TEXT PRIMARY KEY DEFAULT ('evt_' || lower(hex(randomblob(12)))),
  tenant_id       TEXT NOT NULL,
  model_id        TEXT NOT NULL,          -- the client's model identifier
  decision_type   TEXT NOT NULL,          -- e.g. credit_scoring, hiring, medical_diagnosis
  input_hash      TEXT,                   -- SHA-256 of the input (PII-safe)
  output_summary  TEXT,                   -- truncated description of the output
  timestamp       TEXT NOT NULL,          -- when the decision occurred (client-reported)
  metadata        TEXT,                   -- JSON blob for additional fields
  agent_id        TEXT,                   -- partner who referred this tenant (denormalised for speed)
  received_at     TEXT NOT NULL DEFAULT (strftime('%Y-%m-%dT%H:%M:%SZ', 'now')),
  processing_status TEXT NOT NULL DEFAULT 'pending'
                  CHECK(processing_status IN ('pending', 'processing', 'complete', 'failed')),
  FOREIGN KEY (tenant_id) REFERENCES accounts(tenant_id)
);

CREATE INDEX idx_scan_events_tenant_received ON scan_events(tenant_id, received_at);
CREATE INDEX idx_scan_events_status ON scan_events(processing_status) WHERE processing_status = 'pending';
```

### `verification_results` — Output of the Verification Engine

```sql
CREATE TABLE verification_results (
  id              TEXT PRIMARY KEY DEFAULT ('vrf_' || lower(hex(randomblob(12)))),
  event_id        TEXT NOT NULL,
  tenant_id       TEXT NOT NULL,
  result          TEXT NOT NULL CHECK(result IN ('PASS', 'WARN', 'FAIL')),
  checks_run      TEXT NOT NULL,     -- JSON array of check names that ran
  checks_failed   TEXT NOT NULL DEFAULT '[]',  -- JSON array of failed check names
  reason_codes    TEXT NOT NULL DEFAULT '[]',  -- JSON array: [{code, message}]
  latency_ms      INTEGER,           -- how long the engine took
  verified_at     TEXT NOT NULL DEFAULT (strftime('%Y-%m-%dT%H:%M:%SZ', 'now')),
  FOREIGN KEY (event_id) REFERENCES scan_events(id),
  FOREIGN KEY (tenant_id) REFERENCES accounts(tenant_id)
);

CREATE INDEX idx_vrf_tenant_verified ON verification_results(tenant_id, verified_at);
CREATE INDEX idx_vrf_event ON verification_results(event_id);
```

### `vault_records` — Immutable Evidence Vault (hash chain)

```sql
CREATE TABLE vault_records (
  id              TEXT PRIMARY KEY DEFAULT ('vlt_' || lower(hex(randomblob(12)))),
  tenant_id       TEXT NOT NULL,
  sequence_num    INTEGER NOT NULL,        -- monotonically increasing per tenant
  result_id       TEXT NOT NULL,           -- FK to verification_results
  payload_hash    TEXT NOT NULL,           -- SHA-256 of the full verification_result JSON
  chain_hash      TEXT NOT NULL,           -- SHA-256 of (payload_hash + previous chain_hash)
  previous_id     TEXT,                    -- FK to previous vault_records row (NULL for first)
  created_at      TEXT NOT NULL DEFAULT (strftime('%Y-%m-%dT%H:%M:%SZ', 'now')),
  FOREIGN KEY (tenant_id) REFERENCES accounts(tenant_id),
  FOREIGN KEY (result_id) REFERENCES verification_results(id),
  UNIQUE(tenant_id, sequence_num)
);
-- Note: No UPDATE or DELETE operations permitted on this table.
-- Enforced at the application layer. Consider a trigger for belt-and-suspenders.

CREATE TRIGGER vault_no_update
  BEFORE UPDATE ON vault_records
BEGIN
  SELECT RAISE(ABORT, 'vault_records is immutable: updates are not permitted');
END;

CREATE TRIGGER vault_no_delete
  BEFORE DELETE ON vault_records
BEGIN
  SELECT RAISE(ABORT, 'vault_records is immutable: deletes are not permitted');
END;

CREATE INDEX idx_vault_tenant_seq ON vault_records(tenant_id, sequence_num);
```

### `job_queue` — In-process worker queue

```sql
CREATE TABLE job_queue (

  job_type        TEXT NOT NULL,      -- verification, payout, vault_hash, email
  payload         TEXT NOT NULL,      -- JSON
  status          TEXT NOT NULL DEFAULT 'pending'
                  CHECK(status IN ('pending', 'processing', 'done', 'failed')),

  created_at      TEXT NOT NULL DEFAULT (strftime('%Y-%m-%dT%H:%M:%SZ', 'now')),
```

## Migration Execution Order

Run these in order. Each is idempotent (uses `IF NOT EXISTS` or `ADD COLUMN` which fails silently if the column exists).

```sql
-- Migration 001: Extend accounts
ALTER TABLE accounts ADD COLUMN billing_customer_id TEXT;
ALTER TABLE accounts ADD COLUMN billing_subscription_id TEXT;
ALTER TABLE accounts ADD COLUMN run_count_this_month INTEGER NOT NULL DEFAULT 0;
ALTER TABLE accounts ADD COLUMN run_limit_monthly INTEGER NOT NULL DEFAULT 500;
ALTER TABLE accounts ADD COLUMN referral_agent_id TEXT;
ALTER TABLE accounts ADD COLUMN pilot_started_at TEXT;
ALTER TABLE accounts ADD COLUMN trial_ends_at TEXT;

-- Migration 002: Extend users
ALTER TABLE users ADD COLUMN created_by TEXT;
ALTER TABLE users ADD COLUMN last_login_at TEXT;
ALTER TABLE users ADD COLUMN deactivated_at TEXT;
ALTER TABLE users ADD COLUMN deactivated_by TEXT;

-- Migration 003: New tables
CREATE TABLE IF NOT EXISTS scan_events ( ... );
CREATE TABLE IF NOT EXISTS verification_results ( ... );
CREATE TABLE IF NOT EXISTS vault_records ( ... );
CREATE TABLE IF NOT EXISTS compliance_alerts ( ... );
CREATE TABLE IF NOT EXISTS json_policy_maps ( ... );
CREATE TABLE IF NOT EXISTS job_queue ( ... );
CREATE TABLE IF NOT EXISTS partners ( ... );
CREATE TABLE IF NOT EXISTS commissions ( ... );
CREATE TABLE IF NOT EXISTS trust_badge_cache ( ... );
CREATE TABLE IF NOT EXISTS audit_log ( ... );

-- Migration 004: Seed policy maps
INSERT OR IGNORE INTO json_policy_maps ( ... );
```

Store each migration as a numbered file: `src/db/migrations/001_extend_accounts.sql`, etc. Run at startup via a migration runner that tracks which migrations have been applied in a `schema_migrations` table.

## Key Query Patterns

```sql
-- Dashboard summary (always scoped to tenant)
SELECT
  COUNT(*) as total_runs,
  SUM(CASE WHEN result = 'PASS' THEN 1 ELSE 0 END) as pass_count,
  SUM(CASE WHEN result = 'WARN' THEN 1 ELSE 0 END) as warn_count,
  SUM(CASE WHEN result = 'FAIL' THEN 1 ELSE 0 END) as fail_count,
  ROUND(100.0 * SUM(CASE WHEN result = 'PASS' THEN 1 ELSE 0 END) / COUNT(*), 1) as score
FROM verification_results
WHERE tenant_id = ? AND verified_at >= date('now', '-30 days');

-- Vault integrity check (run daily)
SELECT vr.id, vr.chain_hash,
       SHA256(vr.payload_hash || prev.chain_hash) as expected_chain_hash
FROM vault_records vr
LEFT JOIN vault_records prev ON vr.previous_id = prev.id
WHERE vr.tenant_id = ?
  AND vr.chain_hash != SHA256(vr.payload_hash || COALESCE(prev.chain_hash, ''))
ORDER BY vr.sequence_num;

-- Unacknowledged alerts
SELECT a.*, e.model_id, e.decision_type
FROM compliance_alerts a
JOIN scan_events e ON a.event_id = e.id
WHERE a.tenant_id = ? AND a.acknowledged = 0
ORDER BY a.created_at DESC
LIMIT 50;
```
