## Verification 1.2 — Verification Engine reality

**Previous evidence text:** *"`commercialEngine.ts` is referenced in the slice 7 tracker, suggesting some engine layer exists. Whether it does verification-against-policy work as described is unverified."*

**Verdict: ❌ for the brief's Verification Engine. The code exists in `src/engine/` but has no callers and does not execute in any production code path.**

### What's in `src/engine/`

Two files, both well-structured:

- **`verification.ts`** (13.6 KB) — `VerificationEngine` class. Takes a `DomainEvent`, asks `PolicyEngine.getApplicablePolicies(type, env)`, evaluates each policy's `conditions` against the event with seven operators (`eq`, `neq`, `exists`, `contains`, `gt`, `lt`, `matches`), builds per-policy `VerificationResult` records, aggregates to `pass`/`fail`/`inconclusive`, emits `ComplianceAlert` objects on failure, and provides a `buildEvidenceRecord()` that SHA-256-hashes the batch with a `previousHash` for chaining. This is genuinely the shape the brief describes.

- **`policy.ts`** (8.5 KB) — `PolicyEngine` class. Reads `*.json` from `POLICY_TEMPLATE_PATH` (defaults to `./policies/`), validates each rule's shape, indexes by ID, supports `getApplicablePolicies()`, `getPolicy()`, `listPolicies(filter)`, `reload()`, and `watch()` / `stopWatch()` for hot reload. Skips `enabled: false` policies at load time.

Quality-wise, this is some of the most carefully designed code in the upload set. The `FORCE_FAIL_EVENT_TYPES` set on lines 33-36 of `verification.ts` is exactly the override referenced by the `_note` fields in `default.json` — `secret.exposed` and `audit.log_tampered` short-circuit to fail regardless of conditions. The author of the policy JSON and the author of this file were the same person, or coordinated.

### Why it's ❌ anyway

Three findings, in order of severity:

**Finding 2A — No production caller anywhere in the upload set.** A grep across `src/routes/`, `src/services/commercialEngine.ts`, `tests/`, and `policies/` for `VerificationEngine`, `PolicyEngine`, `PolicyEngine.create`, or imports from `'./engine/'` / `'../engine/'` returns **zero results**. The engine is wired to itself; nothing else is wired to it.

**Finding 2B — `pilot-ingest.ts` explicitly defers verification.** The webhook ingest endpoint — the entry point that should feed events into the engine — has a comment block (lines 14-19) stating:

> *Latency contract (per spec): Handler does shape validation + single INSERT + return 202. No synchronous policy evaluation, no anomaly scoring, no downstream calls. Async processing (which writes processed_at) is a separate job — TODO(phase15).*

So events arrive, they're INSERTed into `pilot_decisions`, the endpoint returns 202, and nothing happens. The async processor that *would* call the engine is a TODO. The architecture decision to keep ingest fast is sound; the missing other half is the gap.

**Finding 2C — No tests.** `tests/` contains 10 test files. None reference `VerificationEngine` or `PolicyEngine`. The two files that grep matched (`badge-rotation.test.ts`, `preflight-validate.test.ts`) use "verification" in unrelated senses (badge signature verification; preflight-validating the `validate` middleware). For code that's the heart of the platform's compliance claim, zero unit tests is a significant gap on its own.

## Verification 1.1 — Evidence Vault integrity

**Previous evidence text:** *"SQLite + audit-log tables + R2 backups, but no WORM Object Lock. Hash-chain or signed-record discipline status unclear."*

**Verdict: 🟡** — substantially richer tamper-*evidence* than the gap map credited the platform with, but no tamper-*resistance* (no WORM), and one real design issue.

### What exists (per `audit-coverage.md`)

The audit-coverage doc is a slice-5 deliverable that walked every mutation endpoint and classified its audit behaviour. It documents **two coexisting tamper-evidence philosophies, deliberately designed**:

**Philosophy 1 — `commercial_audit_log` / `role_audit_log` (generic mutation audit).** Used by `users.ts`, `provisioning.ts`, `insurance.ts`. Each row records `actor_user_id`, `entity_id`, `action`, `old_value`, `new_value`. This is the "audit log table" pattern the gap map already understood.

**Philosophy 2 — Per-table HMAC tamper-evidence (domain audit).** Used by every state-changing operation in `commercialEngine.ts`. The state row itself carries `signature` / `chained_hash` / `golden_thread_hash`:

- `commercial_billing_ledgers` — HMAC-signed invoice rows
- `tenant_commercial_subscriptions` — HMAC-signed subscription rows
- `insurance_underwriting_registry` — golden-thread hash chain on policy state
- `underwriting_audit_snapshots` — chained hash linking risk-band recompute history
- `premium_reduction_tokens` — HMAC-signed and chained

The audit-coverage doc is explicit: "this is a deliberate architectural pattern, not an oversight." Confirmed by reading `commercialEngine.ts`: `loadHmacSecret()` refuses to start in production without a secret (line 300+), HMAC verification on subscription reads detects tampering (line 395), tokens carry separate signatures verified at apply time (line 1124+).

The new `VerificationEngine.buildEvidenceRecord()` (lines 287-318 of `verification.ts`) extends this to a SHA-256 hash chain on evidence records: each record includes the `previousHash` from the vault's last record. This is exactly the pattern needed to make the *content* of the vault tamper-evident even on commodity storage.

### What's still missing

**Missing piece A — WORM at the storage layer.** None of the above prevents tampering; it only detects it. A malicious or compromised operator with SQL access can rewrite a row, then run `commercialEngine.generateInvoice` again with synthetic inputs to regenerate the signature, then forge the hash chain by rewriting the next row's `previousHash` too. R2 Object Lock on `caas-lite-backups`, configured for `compliance` mode (not `governance`), would prevent backups from being mutated — which is the difference between "audit catches the tamper" and "tamper can't be hidden by also tampering with backups." The gap map currently marks Object Lock as ❌; still ❌.

**Missing piece B — Atomicity gaps in `commercial_audit_log` writers.** The audit-coverage doc documents 5+ routes where `commercialAuditLog(...)` is called *after* `db.transaction(()=>{...})()` returns. A crash between transaction commit and audit insert leaves the mutation persisted with no audit row. Specifically called out:
- `insurance.ts /policies` (bind)
- `insurance.ts /policies/:id/external` (PATCH)
- `provisioning.ts /` (POST)
- `provisioning.ts /:id/rotate-key`
- `provisioning.ts /:id/tier`

The audit-coverage doc names this "Issue B" and recommends moving audit inserts inside the transaction. This is a slice-6b item per that doc, but it should appear in the Block 3 punchlist for pilot purposes — if even one of these atomicity-gapped audit rows is missing during a pilot, the Trust Badge's "every action was recorded" claim is false on a corner case.

**Missing piece C — Restore drill never run.** Gap map already says ❌. Backups exist via Litestream but have never been restored. Listed in DELIVERABLE_PATH § 3.8 as a Block 3 item; carrying it forward to the punchlist.

**Missing piece D — `caas_evidence.db` second database.** `query-evidence.ts` (now confirmed to be a leftover debug script in `policies/`) references `caas_evidence.db`, distinct from `/data/caas.db`. The audit-coverage doc does not mention this second DB anywhere. **Either it's a development scratch DB that never went to production (so nothing is stored there in prod), or it's a real second store that the disaster-recovery story doesn't cover.** Resolve by checking Fly's volume configuration: if there's only one volume, the second DB doesn't persist in production and `query-evidence.ts` is dev-only. If there are two, the disaster-recovery story extends.

## Verification 1.4 — Trust Badge surface

**Previous evidence text:** *"Schema includes trust_badge_registry and trust_badge_history. Whether the embeddable widget and public read-only page exist is unverified."*

**Verdict: 🟡** — a backend API exists, well-designed, but neither (a) a hosted public read-only vault page nor (b) an embeddable JS widget exists.

### What `src/routes/badge.ts` actually provides

A single endpoint:

```
GET /api/v1/badge/:tenantId?sig=<state_signature>
```

Behaviour (lines 1-220):
- Tenant-public, **not** JWT or API-key authenticated
- Gated by a state-signature passed in the query string — an HMAC over `(tenant_id, badge_state, state_changed_at)` stored in `trust_badge_registry`
- Returns `{badge_state, state_reason, state_changed_at}` as JSON
- Returns 404 indistinguishably for: unknown tenant, missing sig, wrong sig — privacy-preserving
- CORS `*`, cache `public, max-age=30` — embedder-friendly
- Two-secret rotation supported (badge-sync.test.ts in tests/ exists)

This is competent backend design.

### What's missing for "embed on a third-party site"

(a) **Public read-only vault page** — there is no HTML route serving an Evidence Vault view to a public user. The `/api/v1/badge/:tenantId` endpoint returns JSON only. A client clicking through the badge would land on an API endpoint, not a web page they can read. The brief envisions "a read-only view of their Evidence Vault summary" (per `LITE_PILOT_SCOPE.md` § 1). That page does not exist.

(b) **Embeddable widget** — no JS bundle, no `<script src=...>`-able file. The embedder cannot drop a one-liner into their site. They have to write their own JavaScript that fetches the API endpoint and renders the result. For a Lite Pilot client (SMB, likely not technical), this is too much friction.

(c) **Wired to real data** — ✅ when (a) and (b) exist, both will read from the same `trust_badge_registry` and `trust_badge_history` tables the API endpoint already reads.

## Recommendations for Block 2 (pilot scope lock)

Block 2's decisions depend directly on Block 1's findings. Concrete shape now possible:

Recommended wording given verified state: *"Append-only audit trail with HMAC-signed state rows, chained-hash discipline on risk and audit-snapshot tables, and daily off-site replication via Litestream to Cloudflare R2."* This is honest and provable today. **Avoid the word "WORM"** unless P-007 (Object Lock) is committed-to and dated before pilot Week 3. If Block 2 wants the word "tamper-resistant," that requires P-007 first.

**Decision 2 — Verification Engine scope.**
Recommended: pilot the narrower offering ("Compliance Posture Snapshot") in Week 1, expand to "Continuous Verification" once P-001 lands. The engine is well-built; wiring it is the limiting factor. If P-001 can be confirmed for week-of-pilot delivery, full scope; otherwise narrower.

**Decision 3 — Policy coverage.**
Recommended: pilot a single framework Block 3 can deliver well, not multiple frameworks badly. Given the pilot client is SMB (per `LITE_PILOT_SCOPE.md` § 4) and AI-adopting, **EU AI Act is the highest-impact single choice** — it's named in the brief, it's the framework SMB AI users actually want a compliance posture against, and a small initial rule-set (5-10 rules covering data-quality, transparency, and human-oversight clauses) is achievable inside Block 3. GDPR can follow.

**Decision 4 — PDF deliverables.**
Recommended: confirm P-012 sequencing for Week 1 (Initial Risk Scan first deliverable). If P-012 slips, the Initial Risk Scan can be delivered as a markdown report — the deliverable is the *insight*, not the format. Block 2 should write that fallback into the scope explicitly.

**Decision 5 — Integration team scope adjustment.**
Recommended first sprint for the integration team, given Block 1: **P-001 (wire the engine) + P-013 (tests) as the single landing-pad task.** They get to read the best-quality file in the repo (`verification.ts`) to ramp, they ship the load-bearing wiring, and they leave a tested artifact. That's a high-value first sprint.
